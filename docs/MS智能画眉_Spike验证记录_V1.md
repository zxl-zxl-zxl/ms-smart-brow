## 4. S1：Taro 工程初始化与构建

### 验证目标

确认项目可以使用 Taro + React + TypeScript 初始化，并正常构建微信小程序产物。

### 验证结果

- 已建立最小 Taro 工程骨架。
- 已生成首页、相机 Spike 页、保存成功页、个人中心/隐私说明/关于我们占位页。
- `npm run typecheck` 通过。
- `npm run build:weapp` 通过。
- 微信小程序产物输出到 `dist/`。

### 发现问题

初始 Taro 配置中 `designWidth: 375`，但 `deviceRatio` 未配置 `375`，导致编译后 WXSS 出现 `NaNrpx`。

### 处理方式

已在 `config/index.ts` 中增加：

```ts
deviceRatio: {
  375: 2,
  640: 2.34 / 2,
  750: 1,
  828: 1.81 / 2,
}
```

修复后已确认 `dist/` 中不再出现 `NaNrpx`。

## 5. S2：前置相机真机预览

### 验证目标

确认微信小程序真机预览中可以打开前置摄像头。

### 验证结果

- 手机扫码预览后，`camera` 前置摄像头生效。
- 开发者工具连接手机摄像头时曾出现后摄画面，但真机扫码预览前置生效。

### 结论

V1 可以使用微信小程序 `camera` 组件作为实时画眉镜面基础。

### 注意事项

相机相关能力应以手机真机预览为准，不应仅依赖微信开发者工具模拟或摄像头桥接模式。

## 6. S3：拍照保存净图

### 验证目标

确认用户点击保存时，保存到手机相册的是不含辅助线的真实相机画面。

### 验证结果

- `CameraContext.takePhoto` 可用。
- `Taro.saveImageToPhotosAlbum` 可保存到手机本地相册。
- 当前验证中保存的是相机原始画面，不包含屏幕上的辅助线覆盖层。

### 结论

PRD 中“保存净图，不保存辅助线”的关键要求可行。

## 7. S4：相机上方辅助线覆盖层

### 验证目标

确认辅助线能稳定覆盖在 `camera` 画面上方。

### 验证过程

尝试过以下方案：

- 普通 `View` 覆盖层。
- `CoverView` 覆盖层。
- `CoverImage` 加载 SVG/data URI。
- `CoverImage` 加载 PNG/data URI。

### 验证结果

- 修复 `NaNrpx` 后，普通 `View` 覆盖层可显示在相机上方。
- 可显示测试文字、脸部竖线、眼睛水平线、左右眉轮廓线。
- `CoverImage` 内联 SVG/PNG 方案未作为主方案继续推进。

### 结论

V1 主方案可采用普通 `View` 辅助层覆盖 `camera`。

### 备注

当前辅助线只是 Spike 测试用占位线，线条粗细、长度、形态不代表最终设计效果。正式开发时需由 `browEngine` 生成更细、更淡、更贴合的眉形和五官辅助线。

## 8. S5：滑杆微调辅助线

### 验证目标

确认辅助线可通过滑杆进行上下、左右、大小、旋转、透明度调整。

### 验证结果

真机预览中已验证：

- 上下调整有效。
- 左右调整有效。
- 大小缩放有效。
- 旋转角度有效。
- 透明度调整有效。

### 结论

PRD 中“底部滑杆微调辅助线”的交互方案可行。

### 注意事项

当前滑杆调整的是整组辅助线。V1 仍建议保持双眉整体联动调整，不做左右眉单独调整。

## 9. S6：单帧人脸识别与关键点返回

### 验证目标

确认用户点击「开始定标」时，可以基于当前相机帧进行一次人脸识别，并返回可用于后续定位的关键点数据。

### 技术实现

- `camera` 设置 `frameSize='small'`。
- 使用 `CameraContext.onCameraFrame` 持续缓存最新一帧。
- 页面初始化时调用 `Taro.initFaceDetect`。
- 用户点击「开始定标」时，取最新一帧调用 `Taro.faceDetect`。
- 调用参数启用：
  - `enablePoint: true`
  - `enableConf: true`
  - `enableAngle: true`
  - `enableMultiFace: true`

### 验证结果

真机预览中返回结果：

- 识别成功。
- 返回 `106` 个人脸关键点。
- 人脸数为 `1`。
- 返回人脸中心点。
- 返回人脸框，格式已兼容 `width / weight`。
- 真实返回结构主要位于 `faceInfo` 字段。

### 发现问题

Taro 类型定义与微信实际返回结构存在差异：

- 类型定义中字段位于顶层。
- 真机实际返回中主要字段位于 `faceInfo`。
- `detectRect` 宽度字段需要兼容 `width` 和 `weight`。
- 当前测试中置信度和角度可能为空，V1 不应强依赖这两个字段。

### 处理方式

已实现兼容解析逻辑：

- 优先读取 `faceInfo[0]`。
- 其次读取 `faceInfo` 对象。
- 最后兼容顶层字段。
- `detectRect` 兼容 `width / weight`。

### 结论

定标瞬间人脸识别方案可行。V1 可以基于微信端侧 `faceDetect` 返回的 106 个人脸关键点，继续开发脸部定位线和眉形辅助线生成逻辑。

## 10. 当前总体结论

截至当前 Spike，V1 核心技术链路已初步跑通：

- 前置相机可用。
- 相机帧可获取。
- 辅助线可覆盖在相机上方。
- 辅助线可通过滑杆微调。
- 拍照保存净图可用。
- 单帧人脸识别可返回 106 个关键点。

因此，PRD 中的核心方案：

> 手动定标 + 固定辅助线 + 实时相机对齐画眉

具备继续开发的技术可行性。

## 11. 后续建议

下一阶段建议进入以下开发/验证：

1. 建立 `faceAnalyzer` 模块，将当前 Spike 代码从页面中抽离。
2. 建立 `browEngine` 模块，将 106 点关键点转换为脸部中线、眼睛水平线、眉区参考点。
3. 建立正式的辅助线数据结构，例如 `OverlayData`。
4. 用真实关键点驱动当前 `View` 辅助线位置。
5. 将测试辅助线替换为更细、更贴近 PRD 的正式茶棕色线条。
6. 验证不同光线、距离、脸部角度下的识别成功率。
7. 记录失败场景和提示文案。
